13 // This header file will kick in settings for Visual C++ 5 and 6 that will (usually)
14 // result in smaller exe's.
15 // The "trick" is to tell the compiler to not pad out the function calls; this is done
16 // by not using the /O1 or /O2 option - if you do, you implicitly use /Gy, which pads
17 // out each and every function call. In one single 500k dll, I managed to cut out 120k
18 // by this alone!
19 // The other two "tricks" are telling the Linker to merge all data-type segments together
20 // in the exe file. The relocation, read-only (constants) data, and code section (.text)
21 // sections can almost always be merged. Each section merged can save 4k in exe space,
22 // since each section is padded out to 4k chunks. This is very noticeable with smaller
23 // exes, since you could have only 700 bytes of data, 300 bytes of code, 94 bytes of
24 // strings - padded out, this could be 12k of runtime, for 1094 bytes of stuff! For larger
25 // programs, this is less overall, but can save at least 4k.
26 // Note that if you're using MFC static or some other 3rd party libs, you may get poor
27 // results with merging the readonly (.rdata) section - the exe may grow larger.
28 // To use this feature, define _MERGE_DATA_ in your project or before this header is used.
29 // With Visual C++ 5, the program uses a file alignment of 512 bytes, which results
30 // in a small exe. Under VC6, the program instead uses 4k, which is the same as the
31 // section size. The reason (from what I understand) is that 4k is the chunk size of
32 // the virtual memory manager, and that WinAlign (an end-user tuning tool for Win98)
33 // will re-align the programs on this boundary. The problem with this is that all of
34 // Microsoft's system exes and dlls are *NOT* tuned like this, and using 4k causes serious
35 // exe bloat. Very noticeable for smaller programs.
36 // The "trick" for this is to use the undocumented FILEALIGN linker parm to change the
37 // padding from 4k to 1/2k, which results in a much smaller exe - anywhere from 20%-75%
38 // depending on the size. Note that this is the same as using /OPT:NOWIN98, which *is*
39 // a previously documented switch, but was left out of the docs for some reason in VC6 and
40 // all of the current MSDN's - see KB:Q235956 for more information.
41 // Microsoft does say that using the 4k alignment will "speed up process loading",
42 // but I've been unable to notice a difference, even on my P180, with a very large (4meg) exe.
43 // Please note, however, that this will probably not change the size of the COMPRESSED
44 // file (either in a .zip file or in an install archive), since this 4k is all zeroes and
45 // gets compressed away.
46 // Also, the /ALIGN:4096 switch will "magically" do the same thing, even though this is the
47 // default setting for this switch. Apparently this sets the same values as the above two
48 // switches do. We do not use this in this header, since it smacks of a bug and not a feature.
49 // Thanks to Michael Geary <Mike@Geary.com> for some additional tips!
